F Charts — the primary surface

The plots are how the app actually answers a question, so they get their own system, not a coat of theme paint. One categorical set (below) serves every chart and the graph — validated colour-blind-safe on the dark ground (worst adjacent CVD ΔE 8.4). Titles leave the Plotly figure and become page headings; axes and gridlines move onto the palette's chrome tokens; every number is tabular in its column.

v1 — emphasis: context recedes, one line reads
  • Series colour≤ 8 picks (Top 25%, a few chosen cards): the eight slots, one per entity, stable across a re-cut. Past 8: everything grey, hover a legend row to raise it.
  • Line & markersKeep the ADR-0013 read — thin dashed join (asserts no trend) + hollow observation markers — now with a 2px surface ring so overlaps don't turn to mud.
  • Gridlines#2c2c2a hairline, solid (never dashed) — recedes under the data.
  • Axis / ticks#9ca3af → muted #898781, ticks tabular & thousands-comma'd.
  • TitleLeaves the figure; becomes the page's result/section heading in real type, not Plotly's font inside an image.
  • Legend≤ 4 series: direct end-labels. More: legend present, click-to-isolate (Plotly-native, so it survives the gr.Plot hover limit, #78).
  • Range sliderThe head-to-head time filter moves out of the figure (drop the ◀ drag ▶ annotation hack) into a control row above the chart.

G The graph, on the same system

The eight node kinds reuse the eight series slots, so a Deck is the same blue whether it's a dot in the graph or a line in a chart — one categorical vocabulary across the whole app. Grouped views (head-to-head, co-occurrence) tint by group using slots 1–3, the trio that stays distinct even when any two can sit adjacent.

  • Ground#ffffff vis.js default → surface #1c1917. The iframe stops being a white slab and joins the page.
  • NodesUniform dots, name beside the node (the signed rule), quantities on edges. Weighted views size the dot within a bounded range.
  • Node colourThe 8 slots, one per kind — but on a graph any two kinds can touch, so colour is a secondary cue: the label + the on-screen key carry identity, never hue alone.
  • Grouped colourSlots 1–3 (blue / orange / aqua) per player or seed card — the all-pairs-safe trio.
  • EdgesHairline on a border token; tinted to the group in grouped views, neutral otherwise. Labelled quantities stay on the edge.
  • Heightfixed 760 / 700px → responsive, so it's not a phone-eating slab or a desktop letterbox; details panel visible without scrolling.

Open decisions

Everything above is a proposal, not a decision. These are the calls I did not make for you — each changes the work, so they are yours. One at a time when you are ready.

  • Emphasis vs. ADR-0013 colouring. The many-lines fix (grey context + one raised line) changes how the meta/adoption charts use colour — today ADR-0013 colours by rank and leans on the legend. Emphasis is more honest at ~15 lines, but it's ADR territory, so it needs the domain call before it's built. recommend: emphasis
  • Per-point sample-size labels. The performance chart labels every point with "N ev" — useful (a 2-event mean ≠ a 20-event one) but it's the "number on every point" anti-pattern. Keep them, or move to hover and label only the extremes?
  • Accent hue. I kept the app's existing orange (#e26a2c). Keep it, or pick something that reads more like MTG / competitive metagame? recommend: keep
  • Colour-blind canvas cue. The signed rule is uniform dots, so I did not encode kind by shape. A labelled key covers most of it; the on-canvas cue for ~8% of readers (a kind prefix on the label, or a thin ring) is a real trade-off against clutter.
  • Cold start. Land on a drawn default (one real query at startup) or two–three one-click example questions? Both kill the empty canvas; the second teaches the tool better.
  • Type stack. System sans keeps load at zero and deploys clean on the Space. A self-hosted display face would add personality at the cost of bytes and one more served asset — worth it for v1, or later?
